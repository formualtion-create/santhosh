0 Required for any deploy required

Generate a strong session secret — the app refuses to boot in production without one of ≥32 chars:

openssl rand -base64 32

Set it as AUTH_SECRET. That is the only hard requirement; everything else unlocks a specific feature and degrades gracefully (stays simulated / no-ops) when absent.

1 Database → PlanetScale (MySQL)

  1. Create a database in PlanetScale (e.g. pawspair) and a password; copy the connection string.
  2. Switch the active schema to the MySQL variant: 
    cp prisma/schema.mysql.prisma prisma/schema.prisma
  3. Set DATABASE_URL to the PlanetScale string (ends with ?sslaccept=strict).
  4. Push the schema (seed is optional — skip for a clean prod DB): 
    npx prisma db push
npm run db:seed   # optional demo data — SKIP in production
The MySQL schema uses relationMode="prisma" (Vitess has no FK constraints) and @db.Text for long fields — already handled.

2 Payments → Razorpay optional

  1. Create Plans in Razorpay: Fetch (₹499/mo) & Pedigree (₹1,299/mo); copy the Plan IDs.
  2. Add a Webhook → https://YOUR_DOMAIN/api/webhooks/razorpay with events subscription.activated, .charged, .cancelled, .completed. Copy the secret.
  3. Set RAZORPAY_KEY_ID, RAZORPAY_KEY_SECRET, RAZORPAY_WEBHOOK_SECRET, RAZORPAY_PLAN_FETCH, RAZORPAY_PLAN_PEDIGREE.

The Membership page switches from “Demo mode” to live Razorpay checkout automatically; plans activate via the signed webhook.

3 Identity verification → KYC optional

src/app/api/verify/route.ts is the integration point. Replace the simulated approval with your provider’s flow (DigiLocker / Signzy / HyperVerge): redirect to the provider, then flip kycStatus to VERIFIED from the provider’s signed webhook (mirror the Razorpay pattern). Only a masked document reference is stored.

Phone badge: swap the simulated step in /api/trust (step:"phone") for a real SMS OTP (MSG91 / Twilio Verify).

4 File uploads → object storage optional

Local/dev writes images to public/uploads. Serverless hosts (Vercel) have a read-only filesystem, so swap these two routes to upload to S3 / Cloudinary / UploadThing and store the returned URL:

  • src/app/api/pet/photo/route.ts — pet profile photos
  • src/app/api/chat/photo/route.ts — photos shared in chat

Both already validate real image bytes (magic-number sniffing); change only the writeFile(...) call. Recommended: add AI image moderation (AWS Rekognition / Sightengine) at the same point.

5 Email → Gmail SMTP or Resend optional

Transactional email (OTP codes, reset links, welcome emails) lives in src/lib/email.ts. Set one provider:

Option A — Gmail SMTP (simplest; ~500/day)

GMAIL_USER="pawspair@gmail.com"
GMAIL_APP_PASSWORD="xxxx xxxx xxxx xxxx"   # App Password, 2FA on
EMAIL_FROM="PawsPair <pawspair@gmail.com>"

Option B — Resend (scales; recommended)

RESEND_API_KEY="re_..."
EMAIL_FROM="PawsPair <hello@yourdomain.com>"
With neither set, email is skipped and OTP codes show on-screen (demo). Configure SPF / DKIM / DMARC so mail avoids spam.

6 Push notifications + daily cron optional

Web Push — generate keys with npx web-push generate-vapid-keys and set NEXT_PUBLIC_VAPID_PUBLIC_KEY, VAPID_PRIVATE_KEY, VAPID_SUBJECT. Match & message pushes fire automatically (honouring each member’s notification prefs).

Daily “thought of the day” — set CRON_SECRET, then schedule a daily GET to /api/cron/daily-thought with header Authorization: Bearer <CRON_SECRET>. On Vercel, copy deploy/vercel.json to the root.

The cron endpoint returns 503 until CRON_SECRET is set — it’s disabled, not public.

7 Deploy

Vercel (recommended)

  1. Push the project to Git and import it in Vercel.
  2. Add env vars (minimum AUTH_SECRET + DATABASE_URL) in Project → Settings.
  3. Copy deploy/vercel.json to the root for the daily cron.
  4. Deploy — build runs prisma generate && next build automatically.
  5. Set NEXT_PUBLIC_SITE_URL=https://yourdomain.com for canonical/sitemap/OG.

Docker (any host)

docker build -t pawspair .
docker run -p 3000:3000 --env-file .env pawspair

8 Scale & hardening

  • Rate limiting is in-memory (src/lib/security.ts) — move to Redis (Upstash) for multiple instances.
  • Error monitoring — add Sentry. Analytics — GA4 / Plausible / PostHog (cookie-consent banner already gates non-essential cookies).
  • Image moderation on uploads (see §4). Backups — enable on your DB host.

✅ Pre-launch checklist

Must-do

  • Strong, unique AUTH_SECRET set
  • NEXT_PUBLIC_SITE_URL = your real domain
  • Database on PlanetScale; schema pushed
  • Did NOT seed demo data into production
  • Legal pages reviewed by a qualified Indian lawyer
  • Demo logins removed / rotated

Integrations

  • Razorpay keys + plan IDs + webhook — test payment verified
  • Email provider set; SPF/DKIM/DMARC; test OTP + reset + welcome received
  • KYC wired into /api/verify; phone SMS-OTP into /api/trust
  • Uploads pointed at S3 / Cloudinary
  • VAPID keys set; CRON_SECRET + vercel.json at root

Security & ops

  • npm audit clean (currently 0 vulnerabilities)
  • Security headers live (CSP, HSTS) — set in next.config.mjs
  • Image moderation on uploads
  • Rate-limit store → Redis if multi-instance
  • Sentry + analytics added; DB backups enabled
Before you launch: the legal pages (Terms, Privacy/DPDP, Refund, Grievance, Declaration) are solid India-aware templates — have them reviewed and finalised by a qualified Indian lawyer.
